**4**
**(a)**: Das Programm könnte Fehler enthalten die auf nur mit einem Code Review weniger auffällig sind. Es können (Finanzielle-) Schäden durch unerkannte Fehler entstehen.

**(b)**: 
Defekt = Fehlerhafte stelle im Code, Bsp.: Falsche SQL Abfrage. 
Versagen = Der Fehlverhalten zur Laufzeit, Bsp.: Falsche Daten oder erwichbarkeit nicht gewähleistet.

**(c)**: Unentdeckte Fehler können im nachhinein teuer werden. Diesbezüglich gibt es die "Rule of Ten", die besagt das wenn der Fehler im Vorfeld die behebung 10 mal günstiger ist.

**5**
**(a)**: Testen generell zeigt die Anwesenheit von Fehlern auf. Vollständiges Testen ist zwar nicht möglich, aber Testen, am besten besonders Frühzeitig, spart dem Unternehmen nicht nur Kosten sondern auch eine schlechte Reputation. Die einführung von Software Tests ermöglicht eine gleichbleibende Qualität und einen erweiterten Kundenstamm. Zudem ebnet es den weg zu einer richtigen Zertifikat. Am besten sieht man das an dem Beispiel von dem Y2K-Bug, dadurch mussten Unternehmen sehr viel geld in nachträgliche Tests und Korrekturen investieren.

**(b)**: 
Mögliche Fehler: Arbeitstage pro Woche können negativ sein
Defekt: Fehlende Prüfung zur negativität des Ergebnisses
Versagen: Falsches oder ungültiges ergebnis

Die Konsequenzen eines unentdeckten Defekts können sich Rechtlich auswirken.

**(c)**: Das Prinzip 7 ist immer relevant, da nie alle Fehler gefunden werden können.
